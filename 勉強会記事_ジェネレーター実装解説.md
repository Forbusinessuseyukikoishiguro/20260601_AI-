# 🐰 勉強会記事｜ジェネレーターツールの実装解説（新人向け／中堅向け）

ボタンを押すと資料がパッと出てくる「単一HTML生成ツール」。今回はその**最小版（mini-gen）**を題材に、**新人向け**と**中堅向け**に分けて、ソースコードを1行ずつ読み解きます。最後に実物（教育設計ジェネレーター）での確認ポイントとUML（クラス図相当）を載せます。

題材：`mini_gen.html`（入力→生成→出力＋セルフテストだけに絞った学習用）

---

# 第1部｜新人向け：1行ずつ読む

## 0. 全体像（まず地図を持つ）

mini-gen は、たった4つの役割でできています。

1. **入力（INPUT）**：ユーザーが値を入れる（HTMLの `<input>`）
2. **生成（GENERATE）**：入力から文字列（Markdown）を組み立てる
3. **出力（OUTPUT）**：作った文字列を画面に表示する
4. **セルフテスト**：起動時に「壊れていないか」を自分で確認する

> うさうさラーメン店のたとえ🍜：①注文を聞く → ②作る → ③出す → ④味見。当たり前の流れを、そのままコードに分けただけです。

---

## 1. 入力を読む小さな部品

```javascript
function val(id){
  var el = document.getElementById(id);   // HTMLから id で要素を取り出す
  return el ? el.value : "";               // 要素があれば値を、無ければ空文字を返す（安全策）
}
```

- `document.getElementById(id)`：HTMLの中から、その `id` を持つ部品を1つ探します。
- `el ? el.value : ""`：**三項演算子**。「`el` があれば `el.value`、無ければ空文字」。
- なぜ空文字を返す？ → 見つからなくても**エラーで止まらない**ように。これが「落ちないコード」の第一歩です。

## 2. 危険な入力を無害化する（XSS対策）

```javascript
function esc(s){
  return String(s).replace(/[<>&]/g, function(c){
    return {"<":"&lt;", ">":"&gt;", "&":"&amp;"}[c];
  });
}
```

- ユーザーが `<script>` のような文字を入れても、そのまま画面に出すと危険（XSS）。
- `<` `>` `&` を、表示用の安全な表記（`&lt;` など）に置き換えます。
- `String(s)`：数値などが来ても文字列に変換してから処理（これも安全策）。

> 合言葉：**外から来た文字は、まず疑う。**

## 3. 入力をまとめる

```javascript
function collectInput(){
  return {
    course: val("course").trim(),
    audience: val("audience").trim()
  };
}
```

- バラバラの入力欄を、**1つのオブジェクト**にまとめます。
- `.trim()`：前後の空白を削除（「 Linux 」→「Linux」）。
- まとめておくと、あとの関数が `inp.course` のように**短く・読みやすく**書けます。

## 4. 生成の本体（ここが心臓）

```javascript
function generate(inp){
  var course = inp.course || "（コース名なし）";
  var who = inp.audience ? inp.audience + "向け" : "受講者向け";
  var md = "";
  md += "# " + esc(course) + " 設計メモ\n\n";
  md += "- 対象：" + esc(who) + "\n";
  md += "- ゴール：自分の言葉で説明できる\n";
  md += "- 進め方：概要 → 説明 → 演習 → ふりかえり\n";
  return md;
}
```

- `inp.course || "（コース名なし）"`：**未入力でも落ちない**ように既定値を用意。
- `md += ...`：空文字に少しずつ足して、Markdownの文章を組み立てます。
- `\n` は改行。`#` は見出し、`-` は箇条書き（Markdownの記号）。
- **大事なポイント**：この関数は「作るだけ」。画面には触りません。**作る係と出す係を分ける**のがコツ。

## 5. 出力（画面に出す係）

```javascript
function render(md){
  document.getElementById("out").textContent = md;
}
```

- `textContent` に入れると、文字は**そのまま安全に**表示されます（HTMLとして解釈されない）。
- 「作る（generate）」と「出す（render）」を分けたので、テストもしやすくなります。

## 6. ボタンが押されたときの流れ

```javascript
function doGenerate(){
  var inp = collectInput();          // ①集める
  if(!inp.course){                   // ②検証（必須チェック）
    alert("コース名を入れてね");
    return;                          // 中断（早期リターン）
  }
  render(generate(inp));             // ③生成して④表示
}
```

- `if(!inp.course)`：コース名が空なら知らせて `return`。これを**早期リターン**と言います。
- 流れが「集める→検証→生成→表示」と一直線。**読めば分かる**のが良いコードです。

## 7. セルフテスト（味見の自動化）

```javascript
T("生成に見出しが入る", function(){
  return generate({course:"テスト"}).indexOf("# テスト") >= 0;
});
```

- `generate()` を実際に呼んで、結果に「# テスト」が含まれるか確認。
- 含まれれば合格。**人が毎回手で確認しなくて済む**ようにします。
- これが「人ではなく仕組み」。新人が触っても、壊れたらすぐ赤く分かります。

## 8. 初期化（つなぎこみ）

```javascript
function init(){
  document.getElementById("genBtn").addEventListener("click", doGenerate);
  runSelfTests();
}
if(document.readyState === "loading"){
  document.addEventListener("DOMContentLoaded", init);
}else{ init(); }
```

- `addEventListener("click", doGenerate)`：ボタンと処理を**つなぐ**。
- `DOMContentLoaded`：HTMLの読み込みが終わってから動かす（部品が無い状態で触らない）。

**新人向けまとめ**：小さな関数に**役割で分ける**／外の入力は**疑ってエスケープ**／**作ると出すを分ける**／**起動時に味見（テスト）**。これだけで“読めて・壊れにくい”ツールになります。

# 第2部｜中堅向け：設計の意図とUML

新人パートの「動く」を、**保守・拡張できる**へ引き上げる視点です。

## A. 関心の分離（SoC）と単方向の流れ

mini-gen は意図的に **入力 → 整形 → 生成 → 出力** の単方向にしています。

```
[DOM入力] --collectInput--> [Inputオブジェクト] --generate--> [Markdown文字列] --render--> [DOM出力]
```

- `generate()` は **純粋関数に近い**（同じ入力なら同じ出力／DOMに副作用なし）。
  → だからテストが容易（DOM不要で `generate({...})` を直接検証できる）。
- `render()` だけが DOM に触れる。**副作用を1か所に閉じ込める**のが保守性の肝。

実物（教育設計ジェネレーター）では、ここが次のように**拡張**されています。

- `generate()` が `generateMarkdown(inp)` の **switch ルーター**になり、成果物タイプごとの `genCourse / genLesson / …`（16種）へ振り分け。
- `render()` 相当が `renderOutput()` で、Markdown→HTML変換（`mdToHtml`）と各種エクスポート（md/CSV/メモ/html/PDF）に展開。

> 設計の合言葉：**追加は「分岐を1つ増やすだけ」で済む形にしておく。** mini-gen の `generate()` は、実物の `switch` の“種”です。

## B. クラス図（責務の地図・UML相当）

JSなのでクラスは使っていませんが、**責務（モジュール）単位**でUMLのクラス図として表すとこうなります。

```
┌────────────────────┐        ┌────────────────────────┐
│      <<UI>>         │        │     <<Model>>          │
│   InputForm        │        │   InputData            │
├────────────────────┤  reads ├────────────────────────┤
│ + #course: input   │──────▶ │ + course: string       │
│ + #audience: input │        │ + audience: string     │
│ + #genBtn: button  │        └────────────────────────┘
└─────────┬──────────┘                  │
          │ click                        │ 渡す
          ▼                              ▼
┌────────────────────┐        ┌────────────────────────┐
│   <<Controller>>   │ calls  │    <<Service>>         │
│   doGenerate()     │──────▶ │   Generator            │
├────────────────────┤        ├────────────────────────┤
│ + collectInput()   │        │ + generate(inp): md    │
│ + 検証(早期return) │        │ + esc(s): string       │
└─────────┬──────────┘        └───────────┬────────────┘
          │ render(md)                     │ uses
          ▼                                ▼
┌────────────────────┐        ┌────────────────────────┐
│    <<View>>        │        │     <<Test>>           │
│   render(md)       │        │   runSelfTests()       │
│   (DOMに出力)      │        │   T(name, fn)          │
└────────────────────┘        └────────────────────────┘
```

- **UI**：入力欄・ボタン（DOM）
- **Controller**：`doGenerate()`（流れの司令塔・検証）
- **Service**：`generate()` / `esc()`（ロジック。DOM非依存）
- **View**：`render()`（出力＝副作用の出口）
- **Test**：`runSelfTests()`（品質ゲート）

MVCに当てはめると **Model=InputData / View=render / Controller=doGenerate**、そこへ **Service（純粋ロジック）** と **Test** を足した形。役割が一方向に流れるので、依存が循環しません。

## C. シーケンス（実行時の流れ・UML相当）

```
User    InputForm   doGenerate   collectInput   generate   render   DOM(out)
 │  click  │            │             │            │         │         │
 │────────▶│───────────▶│             │            │         │         │
 │         │            │─collect────▶│            │         │         │
 │         │            │◀──inp───────│            │         │         │
 │         │       (course空? ⇒ alertしてreturn)   │         │         │
 │         │            │─generate(inp)───────────▶│         │         │
 │         │            │◀──md────────────────────│         │         │
 │         │            │─render(md)──────────────────────▶│         │
 │         │            │                                   │─set────▶│
 │         │            │                                   │  text   │（表示）
```

## D. テスト容易性（なぜこの分け方か）

- `generate()` がDOMに触れないので、**jsdom不要でも**ロジック単体を検証できる。
- 実物では jsdom で「起動時セルフテストの完了待ち → 出力・CSV・XSS・ダウンロード・ページエラー0」を検証し、**5回連続オール緑**を出荷条件にしています。
- mini-gen の `runSelfTests()` は、その**最小サンプル**。観点（生成・反映・エスケープ）を関数で持つ形は実物と同じ。

## E. 拡張の型（中堅が現場でやること）

新しい成果物を足すときの手順を、mini-gen→実物で対応づけると：

1. `generate()` に分岐を足す → 実物では `genXxx(inp)` を追加し `switch` に1行
2. 入力UIを足す（chip/入力欄）→ `collectInput()` に1項目
3. **必ずセルフテストを1〜2個追加**（生成される・反映される）
4. 商用テスト（jsdom）に観点を追加 → 5回連続緑を確認

> この「**分岐＋テストをセットで足す**」習慣が、16成果物まで破綻なく育てられた理由です。
